## 1\. サンプルデータの整理

### 1.1 データ提供元

株式会社サンエー様よりご提供いただいた、スーパーマーケット業務の実データに基づくサンプルデータ。   
サンエーは沖縄県最大の流通企業であり、参加者にとって身近な存在であることがこのデータの大きな強み。

### 1.2 データ全体像

```

├── card-master/              ... ポイントカードマスター（100件・14KB）
├── customer-master/          ... 顧客マスター（100件・35KB）
```

## 2\. 実行環境

### 2.1 Google Cloud データクリーンルームの活用

サンエー様の実データを取り扱うため、**データ漏洩防止**の観点からGoogle Cloudの**データクリーンルーム**（Analytics Hub \+ BigQueryベースの機密データ共有環境）の活用を前提とする。

**データクリーンルームを使うメリット**:

- 参加者がデータをダウンロード・持ち出しできない環境を構築  
- サンエー様のデータガバナンス要件を満たしつつ、ハンズオンを実施  
- 参加者にとっても「クリーンルームとは何か」を実体験できる学びになる

**想定する構成**:

```
[サンエー様データ] → GCS → BigQuery（データクリーンルーム内）
                                    │
                     ┌──────────────┼──────────────┐
                     ▼              ▼              ▼
               BigQuery SQL    Gemini分析    Looker Studio
              （参加者操作）  （自然言語）   （ダッシュボード）
                     │              │              │
                     └──── 分析結果のみ閲覧可 ────┘
                          （生データの持ち出し不可）
```

**検証ポイント**:

- [ ] Analytics Hub のリスティング作成・参加者へのアクセス権限設定  
- [ ] データクリーンルーム内でのGemini in BigQueryの利用可否  
- [ ] Looker Studioからの接続制限（エクスポート制限の設定）  
- [ ] 参加者用Googleアカウントの発行方法とアクセス期限管理  
- [ ] フォールバック案: データクリーンルームが間に合わない場合の代替構成

##### Phase 2: AIが生成したSQLを読み解く（10分）

#### Step 3: 謎解き分析 — 「答えのある問い」に挑戦（30分）

**目的**: 「なぜ？」を考え、仮説を立ててデータで検証する分析の醍醐味を体験する

データ分析の本質は「数字を出すこと」ではなく「なぜ？を考えること」。 ここでは、あえて**答えが分かっている問い**を出題し、参加者がデータから答えにたどり着くプロセスを体験する。 AIに聞いても良いし、SQLを自分で書いても良い。手段は問わない。

##### 謎解き Q1:「元旦の売上が突出して高いのはなぜ？」

**ヒント**: まず日別売上を出してみよう

```sql
-- AIに聞く場合: 「日別の売上合計と取引数を出してください」
-- 自分で書く場合:
SELECT
  usage_date,
  COUNT(DISTINCT CONCAT(card_no, store_code, trans_no)) AS txn_count,
  SUM(tax_included_amount) AS total_sales,
  SAFE_DIVIDE(SUM(tax_included_amount),
    COUNT(DISTINCT CONCAT(card_no, store_code, trans_no))) AS avg_per_txn
FROM `meetup8.customer_sales_details`
GROUP BY usage_date
ORDER BY usage_date;
```

**答え**: 1月1日は売上64,037円で月間最高。客単価3,370円も最高。**お正月の買い出し（おせち材料・お年始用品・正月食材）で1回あたりの購入額が跳ね上がる。** 正月三が日は通常の1.5倍以上の客単価。

**深掘り**: 「正月三が日だけに売れた商品が278種類もある。普段は売れない正月限定商品が動いている。」

##### 謎解き Q2:「1月28日（土曜日）の売上が極端に低いのはなぜ？」

**ヒント**: 曜日別の売上パターンを見てみよう

```sql
-- AIに聞く場合: 「曜日ごとの売上合計を教えてください」
-- 自分で書く場合:
SELECT
  FORMAT_DATE('%A', PARSE_DATE('%Y%m%d', usage_date)) AS day_of_week,
  SUM(tax_included_amount) AS total_sales,
  COUNT(*) AS item_count
FROM `meetup8.customer_sales_details`
GROUP BY day_of_week
ORDER BY total_sales DESC;
```

**答え**: 日曜日が189,006円で最も高く、土曜日は111,404円と意外と低い。特に**1/28(土)は月間最低の12,147円**。これは前日1/27(金)に月間最多の23取引（49,524円）があり、**金曜に買い溜めした翌土曜は来店が減る**パターン。沖縄では日曜の家族連れまとめ買いが主流。

**「土曜が一番売れるはず」という思い込みが、データで覆される瞬間。** これがデータ分析の醍醐味。

##### 謎解き Q3:「一番大量に買い物をしたお客さんはどんな買い方をしている？」

**ヒント**: バスケット分析をしてみよう

```sql
-- AIに聞く場合: 「1回の買い物で一番多くの商品を買った取引を教えてください」
-- 自分で書く場合:
SELECT
  card_no,
  store_code,
  usage_date,
  COUNT(*) AS item_count,
  SUM(tax_included_amount) AS basket_total
FROM `meetup8.customer_sales_details`
GROUP BY card_no, store_code, trans_no, usage_date
ORDER BY item_count DESC
LIMIT 5;
```

**答え**: 最大バスケットは**1/19（木）店舗5040で38品・9,756円**。1品あたり平均256円で、日用品・食品をまとめ買いしていると推測。一方、最高額バスケットは**1/1（元旦）店舗0723で37品・17,349円**。品数は近いのに金額は1.8倍 — 正月は単価の高い商品を買っている。

##### 謎解き Q4:（発展）「ポイントをたくさん持っている人は、本当にたくさん買い物をしている？」

```sql
SELECT
  c.card_no,
  c.total_point AS current_points,
  c.cumulative_issued_point AS lifetime_points,
  COUNT(*) AS purchase_count,
  SUM(s.tax_included_amount) AS total_spent
FROM `meetup8.card_master` c
JOIN `meetup8.customer_sales_details` s ON c.card_no = s.card_no
GROUP BY c.card_no, c.total_point, c.cumulative_issued_point
ORDER BY total_spent DESC
LIMIT 10;
```

**答え**: 必ずしも一致しない。ポイント残高が多い人は「貯める派」、少ない人は「使う派」。購入額が多いのにポイント残高が少ない人は、こまめにポイントを使っている。**顧客行動の多様性**がデータから見える。
